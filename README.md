# ☁️ Real-Time Weather Analysis Pipeline:Azure & Power BI

## 📌 Project Overview
This project is an end-to-end cloud data pipeline built entirely within the Azure ecosystem. It automates the extraction of live weather data for Dublin, London, New York, and Dubai, orchestrating its movement from raw extraction to a structured data warehouse, and finally into an interactive dashboard. 

The primary goal was to step away from basic local scripts and build a highly secure, enterprise-grade ETL workflow using Azure's native integration and storage tools.

## 🏗️ Architecture & Stack
* **Data Source:** Open-Meteo API
* **Ingestion/Compute:** Python (Pandas, Requests)
* **Data Lake (Raw):** Azure Blob Storage
* **ETL Orchestration:** Azure Data Factory
* **Data Warehouse:** Azure SQL Database
* **Business Intelligence:** Microsoft Power BI
* **Security:** Azure Managed Identities & RBAC

## ⚙️ How It Works (The Pipeline)

**1️⃣ Data Extraction**
`weather_api.py` pings the Open-Meteo API, cleans the JSON payload, and formats it into a localized CSV file containing climate metrics (temperature, windspeed) and timestamps.

**2️⃣ Data Storage (The Lake)**
The extracted CSV is loaded into an **Azure Blob Storage** container, acting as the raw staging area.

**3️⃣ Integration & ETL**
An **Azure Data Factory** pipeline is triggered to securely copy the data from Blob Storage into the database. This layer relies entirely on Managed Identities, ensuring no credentials or connection strings are hardcoded.

**4️⃣ Structured Warehousing**
The data lands in an **Azure SQL Database** table (`WeatherData`). Strict schema enforcement guarantees that data types (like floats for temperature) are perfectly maintained for downstream analysis.

**5️⃣ Visualization**
**Power BI** connects directly to the Azure SQL Database to generate an interactive dashboard, comparing live climate KPIs across the four global cities.

## 🐛 Key Challenges Solved

**1. Strict Schema Failures in ADF**
* **Cause:** Azure Data Factory pipelines fail if raw CSV data types don't perfectly match the destination SQL schema (e.g., passing a blank string into a `FLOAT` column).
* **Fix:** Implemented strict data-type enforcement and cleaning within the initial Python extraction script before the data ever hits the cloud.

**2. Secure Authentication**
* **Cause:** Passing database credentials in plain text is a massive security vulnerability in enterprise pipelines.
* **Fix:** Configured Azure Role-Based Access Control (RBAC) and Managed Identities, restricting access so services only have the exact permissions required to run the pipeline.
